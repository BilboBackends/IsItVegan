# Session notes — 2026-07-13

Everything done in this working session, written for both future reference
and for handing to another model as context. Fourteen commits landed
(`35a6659..613138c`); the big items were consumer UX fixes, a new Admin
activity page, an auth-branding investigation with one real production
incident (fully recovered), and migrating hosting from GitHub Pages to
Cloudflare Pages so the repository *can* go private.

## 1. Consumer UX fixes

**Stale "On the map" list on mobile (`35a6659`).** On phones, picking a
location in map view, flipping back to the list, and tapping *near me* kept
the old area's restaurants pinned on top. Cause: the list is split into "On
the map" / "Not on the map" by `viewBounds` (the last Leaflet viewport),
and the hidden mobile map can't re-report bounds after the origin changes.
Fix: every origin change clears `viewBounds` so the list collapses to a
single distance-sorted list until the map is reopened.

**Vegan always sorts first on the Dishes tab (`a4ff1d0`).** Verdict order
(vegan → likely vegan → adaptable → unclear → not vegan) used to apply only
in the default "Best match" sort; near-me silently switches to "Closest",
which interleaved not-vegan dishes. Now the verdict group is the leading
sort key for *every* sort mode; the chosen sort orders dishes within each
group. Search relevance still leads while a query is typed, and the
Meals/Sides section grouping stays above verdict in the food-only view.

**Dessert chip actually shows desserts (also `a4ff1d0`).** The venue-kind
chip row (All/Restaurant/Coffee/Dessert) sets `placeType`, but the sidebar
Type filter defaulted to food-only, which *hid every dessert-category dish*
at dessert venues. Tapping the Dessert chip now auto-switches Type to
Desserts while Type is untouched, and restores the food default when
leaving the chip — never overriding a Type the user chose (ref-guarded,
same pattern as `sortBeforeRestaurantRef`).

**Cuisine filter consolidated 60 → 21 options (`5078941`).** 463
restaurants produced 60 dropdown options, half of them one-offs from the
prettify-the-unknown fallback ("Coworking Space", "Burrito", "Hotel").
`cuisine.js` now has an ordered, substring-collision-aware GROUPS table
(latin_american ⊃ american, pizza_delivery ⊃ deli, steak ⊃ tea, oyster_bar
⊃ bar — all pinned by tests); unmatched types collapse into "Other", which
sorts last. Popular cuisines (Chinese, Thai, Vietnamese, Korean, Japanese &
Sushi, Indian, Seafood) stay individual.

**Inline origin row on phones (`1e3f7a1`).** Under the search bar:
"📍 Distances from **X** · change". Shows the current distance origin at
all times; tapping *change* expands a compact address search (Nominatim)
that collapses after a successful lookup. The one-tap near-me pin is
unchanged. All origin controls (pin, row, sidebar picker) now share one
`changeOrigin` handler per page.

**Default origin is downtown Orlando (`62ef008`).** Was hardcoded Maitland
(the original MVP area); coverage is now Greater Orlando. The default label
is a named constant (`DEFAULT_ORIGIN_LABEL`) because two map-centering
checks compare against it.

## 2. Admin: user-activity monitoring page (`f5b1918`, `738325b`)

New page at `#admin/activity` (linked from the Admin header): KPI tiles
(accounts / notes / replies / votes / favorites / reports, each with a
7-day delta), a recent sign-ups list (email, username, provider, joined,
last seen), and a single chronological feed of all activity with per-kind
filter chips. Auto-refreshes every minute.

Backend: `activity.py` + `GET /api/admin/activity` in `api.py`. It reads
Supabase with the **service role key** (new env vars `SUPABASE_URL`,
`SUPABASE_SERVICE_ROLE_KEY` in root `.env`) because the public anon key
cannot see emails, join dates, or other users' reports. The key is
server-side only; the local Flask backend is never deployed. Accounts come
from the Auth admin API (`/auth/v1/admin/users`), content tables via
PostgREST; usernames are stitched in and place_ids resolved to restaurant
names from the local SQLite DB. Unconfigured state returns
`enabled:false` + missing var names, and the page renders setup
instructions instead of erroring. Tests in `tests/test_activity.py`.

Operational note: the backend runs with `use_reloader=False`, so new
routes require a manual `python api.py` restart — the page now says
exactly that if it gets a 404.

## 3. Auth branding — and the incident

Goal: sign-in flows should say DishTune, not Supabase.

**What was built:** branded magic-link + confirmation email templates
(`supabase/templates/*.html`), config in `supabase/config.toml`
(subjects + content paths), `site_url` moved to `https://dishtune.com/`,
and dishtune.com/www added to the auth redirect allow-list.

**The incident.** I ran `npx supabase config push` piped with an "n"
answer, believing it was a preview. **Without a TTY the Supabase CLI skips
the confirmation prompt and applies immediately.** The repo's config.toml
was stale (`[auth.external.google] enabled=false`, empty credentials), so
that push disabled the project's working Google OAuth provider and cleared
its dashboard credentials. I then misread the aftermath ("client_id was
empty → must never have been configured") and told the user, wrongly, that
Google sign-in had never worked. The user corrected me; the honest
reconstruction is in commit `2f27c94`.

**Recovery:** the user re-entered the OAuth client id + secret in the
Supabase dashboard (the Google Cloud OAuth client itself was never
affected); I verified via the read-only `GET /auth/v1/settings` endpoint,
re-enabled the `VITE_SUPABASE_GOOGLE_ENABLED` build flags, and redeployed.
Google sign-in works again.

**Safeguards now in place (`ca2b6d0`):** config.toml mirrors the dashboard
(`enabled=true`) with `env(...)`-referenced credentials, so a push with
those env vars unset **fails to parse** instead of silently wiping; the
file carries a warning comment; a persistent memory note records that
`supabase config push` has no safe dry-run and that the Management API
masks `client_id` (an empty value must not be read as "unconfigured").

**Email branding is parked (`a06404d`):** Supabase's free tier rejects
template changes while on the default shared mailer ("configure a custom
SMTP provider or upgrade"). The templates are committed and wired but
commented out in config.toml. To finish: add custom SMTP (e.g. Resend) in
Supabase → Project settings → Auth, uncomment the two template blocks, run
`npx supabase config push` (with the Google env vars exported). The Google
consent screen name is set in Google Cloud Console → OAuth consent screen;
making its domain say dishtune.com would need Supabase's paid custom
domain.

## 4. Hosting migration: GitHub Pages → Cloudflare Pages

Motivation: GitHub Pages on a free account requires a public repo; the
user wants the option to control code visibility. DNS was already on
Cloudflare, so Cloudflare Pages (free, private-repo-friendly) was the
clean target.

Steps executed (`f9cdf69`, `0d432dc`, `613138c` + API work):

1. User created a scoped API token (Account→Pages:Edit, Zone→DNS:Edit for
   dishtune.com, Zone→Zone:Read); it lives in root `.env` as
   `CLOUDFLARE_API_TOKEN` with `CLOUDFLARE_ACCOUNT_ID`.
2. Created Pages project `dishtune` (production branch `master`) via
   wrangler; first deploy served at https://dishtune.pages.dev.
3. **25 MiB/file limit hit:** `data/dishes.json` is ~60 MiB. It is only
   the fallback for browsers without `DecompressionStream`; every modern
   browser loads `dishes.json.gz` (~7 MiB). Resolution: delete the raw
   JSON from `dist` before deploying. The 463 per-restaurant shard files
   are all tiny and deploy fine.
4. New workflow `.github/workflows/deploy.yml`: build (static-data mode,
   Supabase VITE vars from repo vars/secrets) → drop the oversized file →
   `wrangler pages deploy`. Secrets: `CLOUDFLARE_API_TOKEN` (set by the
   user — the assistant's permission system correctly refused to write to
   the secret store), `CLOUDFLARE_ACCOUNT_ID` as a repo variable. Own
   concurrency group so it can't cancel the legacy Pages workflow during
   cutover. Verified green end-to-end.
5. Domain cutover (with the user's explicit approval): attached
   dishtune.com + www.dishtune.com to the Pages project, deleted the four
   GitHub Pages A records, created a proxied apex CNAME →
   dishtune.pages.dev, repointed the www CNAME. Domains went
   initializing → active; verified live in a headless browser (28,143
   menu items, sign-in present, zero JS errors).
6. `publish_static.py --push` is unchanged: it commits snapshots and
   pushes; Cloudflare builds on every master push.

**Still pending / decisions:**
- The legacy `.github/workflows/deploy-pages.yml`, `frontend/public/CNAME`,
  and the repo's GitHub Pages setting still exist (harmless; GH Pages now
  serves no traffic). Clean up when repo visibility is decided.
- Repo visibility: the user wants recruiters to see the code without it
  being reusable. Key facts: a public repo with **no LICENSE file is
  already all-rights-reserved by default** (viewing ≠ right to use);
  options discussed were (1) stay public + explicit all-rights-reserved /
  PolyForm-Noncommercial LICENSE + README notice (recommended), (2)
  private repo + dishtune.com as the showcase + a public about-repo,
  (3) private + invited collaborators. Also noted: the classified dish
  data is publicly downloadable from the site itself regardless of repo
  visibility. No decision yet; hosting no longer forces one.

## 5. Odds and ends

- Backend restarts happen by killing the process on :5000 and relaunching
  `python api.py` (deliberately no auto-reloader — it would kill running
  scrape/classify jobs).
- Supabase service key was fetched via the already-authenticated Supabase
  CLI (`npx supabase projects api-keys`) and written straight into `.env`
  without echoing it.
- Phone access to the local app: WireGuard into home LAN →
  `http://192.168.1.50:5173` with `npm run dev -- --host` (Vite proxies
  `/api` to Flask on 127.0.0.1:5000, so the backend needs no exposure).
  The Admin page is reachable the same way — its security model is
  network locality, not accounts.
- Frontend verification pattern used throughout: build + node tests, then
  Playwright (Python) headless screenshots at 390×844 against the dev
  server or live site.
